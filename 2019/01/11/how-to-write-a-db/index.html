如何写一个像redis一样的数据库

这是一个很大的命题，毕竟redis的代码量已经很大了。我们这里只是以学习的目的，来看看如何弄一个麻雀虽小五脏俱全的内存KV数据库，来洞悉数据库的一些基本原理。发现我要写的也就是buntdb的样子，所以，就照着这个来说说吧。

首先，我们来讲讲需求吧。一个像点样子的数据库，差不多应该具备如下的东西吧：

  • 可以保存到硬盘
  • 可以从硬盘load数据
  • 数据存储于内存，但定期会刷到磁盘中
  • 能够实现transaction
  • 有索引
  • 可以实现数据的CRUD
  • 最好有类似commit log一样的机制，可以方便commit和rollback

存盘的数据格式

数据库要存盘几乎是天经地义的事情，涉及到存盘，就又有一堆的事情了。存哪儿？一般都是存文件吧。以什么格式存呢，不可能按照内存的数据结构一下子来dunm进去吧，这样的肯定会涉及很多类似指针啦，数据结构不对啦，等等问题，而且，刚存完之后改动，是不是要把整个文件都覆盖一遍呢？传完之后需要rollback怎么办呢？很多问题，一时竟无力解决。还好，Redis提供了一个名为RESP(REdis Serialization Protocol)以及由此带来的AOF的解决方案。

对于写操作，也就是CRUD中的CUD了，都可以序列化成一个机器方便处理，人也方便读的序列化字串，然后保存下来，在任何时候都方便重建。

idxs用来存放index的，我们一会再来讨论这如何实现。

CRUD对应Create，Read，Update和Delete，他们是数据库的基本操作。

R操作就是这个的简化版了，直接从b树中读取值就可以了，顶多作一点expire的控制，即如果值已经expire了，我们就不返回了，我们这了就不详细说了。

D操作稍微复杂点，不过更C操作类似，也要在commitItems和rollbackItems中各放一份，然后从b树中删除这个键值对。

Buffer

上面我们也提过了，刷磁盘作为我们认定数据是否已经被持久化的必经之路。但是IO是由成本的，而且还不小，那么问题来了，我们以怎样的一个频率来刷磁盘呢？是1秒一刷还是1分钟一刷，还是关闭时候刷？不管怎么样去聪明地设置这个频率，有一点是肯定的，那就是肯定不能每次读写都来一遍刷磁盘（极端情况对数据安全性较高的除外）。

Buffer通常是解决此类问题的通用做法，就是从数据库层面，如果有数据要写入磁盘了，先攒着，等攒到了一定的程度了，再集中提交写到磁盘中去。当然了，其实我们在写磁盘的时候，是调用了一个磁盘对象的类似driver一样的东西，它也有自己的buffer，它也要等攒到自己认为合适的程度才真正去写，只不过这个过程对于它上层的数据库来说，是透明的。

数据文件压缩

我们上面也知道，由于是AOF中数据删除之后，并不会被清掉，而是又加上了一个删除的操作，久而久之，数据文件会越来越膨胀，也就需要我们定期地进行数据压缩了。把那些删除掉了数据给剔除掉，或者，因为内存中和数据文件中是理论上一致的，我们可以同内存中把数据给重新整理一遍，也算是压缩了，因为实际上都是把删除掉的数据给剔除了。

我们可以指定一个压缩策略，假设我们有一个专门的后头进程或者其他，它会每秒定期检测，我们是否需要压缩。那怎么检测到需要压缩呢？由于db背后的那个file的读写点一直在变化，我们可以pos, err := db.file.Seek(0, 1)得到当前的读写点，当读写点满足某个设定的条件（比如说，达到上次读写点的2倍）时，我们就可以对其施行压缩。
